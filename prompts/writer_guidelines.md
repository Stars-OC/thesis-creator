# 论文编写提示词指南

> 本文档用于指导 AI 生成高质量的学术论文内容

---

## 两阶段写作法（核心策略）⭐

> 此方法参考 `scientific-writing` SKILL.md，是降低 AI 模板化输出的核心策略

### 核心原则

**绝不直接输出段落**，而是「先提纲 → 再成文」。此策略可有效：
- 降低 AI 模板化输出倾向
- 增加文本结构多样性
- 便于用户在要点阶段介入调整
- 与 Step 6（AIGC 人性化）形成**前后夹击**策略

### Stage 1：段落级要点规划

对每一章节，**先生成结构化要点列表**，不直接写正文：

- 每段的核心论点（1 句话）
- 计划引用的文献（作者+年份）
- 支撑论据（数据/案例/理论）
- 段落间的逻辑衔接关系

**示例输出：**

```
第三章 大数据在精准营销中的应用
├─ § 3.1 用户画像构建
│  ├─ 要点：大数据技术使用户画像从静态标签转向动态行为建模
│  ├─ 引用：张三（2023）用户行为分析框架、李四（2022）画像精度研究
│  ├─ 论据：某电商平台画像准确率提升 27%（具体数据）
│  └─ 衔接：→ 引出 § 3.2 精准推荐
│
├─ § 3.2 精准推荐算法
│  ├─ 要点：协同过滤与深度学习结合提升推荐精度
│  ├─ 引用：王五（2024）混合推荐模型
│  ├─ 论据：A/B 测试点击率提升 15.3%
│  └─ 衔接：→ 过渡至效果评估
│
└─ § 3.3 营销效果评估
   ├─ 要点：建立多维度评估体系量化营销效果
   ├─ 引用：赵六（2023）营销效果测评指标
   ├─ 论据：ROI 提升案例对比
   └─ 衔接：→ 承接下一章
```

**用户确认环节**：Stage 1 输出后，等待用户确认或调整，再进入 Stage 2。

### Stage 2：要点扩展为段落

将上述要点转为连贯的学术段落，遵循以下规则：

1. **变换句式结构**：禁止连续 3 段使用相同的「总分」结构
2. **句长波动**：目标句长标准差 > 10，模拟人类写作节奏
3. **自然引用**：将引用融入句子中，而非堆砌在段末
4. **段首多样性**：相邻段落首词不得相同

**扩展示例（基于上述要点）：**

> 用户画像作为精准营销的基础，其构建方式正经历从静态标签到动态行为建模的转变。张三（2023）提出的用户行为分析框架，为这一转变提供了理论支撑。实践中，某电商平台通过引入实时行为数据，使画像准确率提升了 27%。基于此，精准推荐算法得以进一步优化。协同过滤与深度学习的结合，成为当前推荐系统的主流方案。王五（2024）的混合推荐模型研究表明，这种组合方式在 A/B 测试中实现了 15.3% 的点击率提升。

### 与 AIGC 降低的协同效果

两阶段写作法天然产生更「人类化」的文本：
- Stage 1 的规划过程引入了结构随机性
- Stage 2 的逐点展开避免了 AI 一次性生成的模板化

## 一、整体风格要求

### 1.1 学术正式语体

**推荐表达**：
- 「研究表明…」
- 「据统计分析…」
- 「笔者认为…」
- 「从实证角度来看…」
- 「基于上述分析…」
- 「现有文献表明…」

**禁用 AI 特征词汇**：
| 禁用词汇 | 替代方案 |
|---------|---------|
| 此外 | 另外、同时、与之相伴 |
| 值得注意的是 | 需要关注的是、值得关注的是 |
| 不可否认 | 必须承认、诚然 |
| 综上所述 | 总体而言、概括来说 |
| 首先…其次…最后 | 直接陈述要点，或用「一方面…另一方面」 |
| 具有重要意义 | 意义重大、作用显著、价值突出 |

### 1.2 论证结构多样性

避免每段都是「总分」结构，穿插使用以下方法：

| 结构类型 | 适用场景 | 示例 |
|---------|---------|------|
| 演绎法 | 理论推导 | 从一般原理推导具体结论 |
| 归纳法 | 数据分析 | 从具体案例归纳一般规律 |
| 对比法 | 文献综述 | 对比不同学者观点 |
| 因果法 | 问题分析 | 分析原因→推导结果 |
| 递进法 | 深入探讨 | 层层深入、逐步推进 |

## 二、段落写作规范

### 2.1 段落结构

每个段落应包含：

```
[主题句] 明确表达本段核心观点（20-40字）
[展开句] 提供论据、数据、案例支持（100-200字）
[小结句] 总结本段，呼应主题或承上启下（20-40字）
```

**段落长度**：150-300 字/段

### 2.2 段落首句多样化

避免每段都以相同的句式开头：

| 避免 | 建议 |
|-----|-----|
| 每段都是「XX是…」 | 穿插使用疑问句、引用句、数据句 |
| 每段都是「随着…」 | 使用「当前…」「已有研究表明…」 |
| 每段都是「在…背景下」 | 直接切入主题，或用案例开头 |

## 三、引用规范

> ⚠️ **重要**：编写参考文献时，**必须调用** `prompts/reference_format_gbt7714.md` 获取完整的 GB/T 7714-2015 格式规范。
>
> ⚠️ **更重要**：**必须加载** `prompts/reference_citation_prompt.md` 和 `workspace/references/verified_references.yaml` 文献池！
> **所有引用必须来自已验证文献池，禁止自行编造任何文献信息！**

### 3.0 引用硬约束 ⭐ **NEW**

> **以下规则为硬性约束，违反即为严重错误！**

| 禁止行为 | 说明 | 违规后果 |
|----------|------|----------|
| ❌ **禁止编造文献标题** | 不能虚构论文标题 | 触发重生成 |
| ❌ **禁止编造作者姓名** | 不能使用「张三」「李四」等占位符 | 触发重生成 |
| ❌ **禁止编造期刊名称** | 不能虚构期刊或会议名 | 触发重生成 |
| ❌ **禁止编造 DOI** | 不能生成虚假 DOI 号 | 触发重生成 |
| ❌ **禁止使用未验证文献** | 所有引用必须来自 verified_references.yaml | 标记为「未验证」 |
| ✅ **必须包含 DOI 链接** | 格式：`[DOI](https://doi.org/xxx)` | 缺少则标记 |

**文献池使用流程**：

```
1. 写作前加载 workspace/references/verified_references.yaml
2. 分析段落关键词
3. 从文献池检索匹配文献
4. 选取相关度最高的 1-3 篇
5. 格式化为 GB/T 7714 含 DOI 链接
6. 如果文献池无匹配文献 → 标注「⚠️ [需搜索补充]」
```

**DOI 链接格式要求**：

```markdown
✅ 正确格式：
[1] Lewis P, Perez E, Piktus A, et al. Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks[C]//NAACL 2020. 2020. [DOI](https://doi.org/10.18653/v1/2020.naacl-main.13)

❌ 错误格式（缺少 DOI 链接）：
[1] Lewis P, Perez E, Piktus A, et al. Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks[C]//NAACL 2020. 2020.
```

**中文文献无 DOI 的处理**：

```
[1] 张三, 李四. 某中文期刊论文[J]. 中文期刊, 2022, 15(3): 45-52. ⚠️ 需用户确认

（后续需用户从 CNKI 补充 DOI 或手动核实）
```

---

### 3.1 引用频率

- **最低要求**：每千字至少 2 个文献引用
- **推荐频率**：每千字 3-5 个文献引用
- **分布要求**：各章节均有引用，不要集中在某一章
- **单句硬约束**：每句话最多引用 1 篇文献；相邻句子可分别引用不同文献

### 3.2 引用格式（GB/T 7714-2015）

**严格遵循 GB/T 7714-2015《信息与文献 参考文献著录规则》**。

详细格式请查阅 `prompts/reference_format_gbt7714.md`，以下为快速参考：

**期刊论文[J]**：
```
[序号] 作者. 题名[J]. 刊名, 年, 卷(期): 页码.
```
示例：[1] 张三, 李四. 大数据技术研究[J]. 管理科学学报, 2023, 26(3): 45-52.

**学位论文[D]**：
```
[序号] 作者. 题名[D]. 保存地: 保存单位, 年份.
```
示例：[2] 李明. 推荐算法研究[D]. 北京: 清华大学, 2022.

**专著[M]**：
```
[序号] 作者. 题名[M]. 出版地: 出版者, 出版年: 页码.
```
示例：[3] 刘国钧. 图书馆目录[M]. 北京: 高等教育出版社, 1957: 15-18.

**会议论文[C]**：
```
[序号] 作者. 题名[C]//编者. 会议名, 会议地点, 时间: 页码.
```

**科技报告[R]**：
```
[序号] 作者. 报告题名[R]. 报告地: 报告单位, 年份.
```

**标准[S]**：
```
[序号] 标准编号, 标准名称[S].
```
示例：[4] GB/T 7714-2015, 信息与文献 参考文献著录规则[S].

**专利[P]**：
```
[序号] 专利申请者. 专利题名: 专利号[P]. 公告日期.
```

**报纸文章[N]**：
```
[序号] 作者. 题名[N]. 报纸名, 出版日期(版次).
```

**电子文献[EB/OL]、[J/OL]等**：
```
[序号] 作者. 题名[文献类型/载体标识]. (发布日期)[引用日期]. URL.
```
示例：[5] 萧钰. 出版业信息化[EB/OL]. (2001-12-19)[2002-04-15]. http://www.example.com/article.html.

### 3.3 作者姓名著录规则

**中文作者**：
- 不超过 3 人：全部著录
- 超过 3 人：著录前 3 人后加「, 等」

**欧美作者**：
- 姓全大写，名缩写
- 姓在前，名在后
- 超过 3 人：著录前 3 人后加「, et al.」

示例：
```
EINSTEIN A.
EINSTEIN A, BOHR N, PLANCK M, et al.
```

### 3.4 正文引用标注

**标注位置**：引用标注置于所引内容最末句的右上角。

**标注方式**：
| 引用情况 | 标注方式 | 示例 |
|---------|---------|------|
| 单篇文献 | [序号] | [1] |

> **硬约束**：禁止同一句出现多个引用编号（如 `[1,3]`、`[1-3]`、`[1][2]`）。需要多文献支撑时，拆分为多句并分别引用。

### 3.5 引用方式

**直接引用**：
> 张三（2023）指出：「大数据技术的发展为精准营销提供了新的可能性。」

**间接引用**：
> 根据张三（2023）的研究，大数据技术为精准营销带来了新的机遇。

**多文献并列**：
> 众多学者对此进行了研究（张三, 2023; 李四, 2022; Wang, 2021）。

## 四、章节写作要点

### 4.1 摘要（逻辑三段，篇幅约当前页面 2/3）

> **硬约束**：摘要需覆盖三段逻辑（背景与目标、主要内容与技术路线、成果与意义），中文篇幅建议约占当前页面 2/3；不强制自然段数量必须等于 3。

**第一部分：背景和本文目标**
- 说明研究问题领域、现实需求和本文的研究目标
- 篇幅约占摘要的三分之一

**第二部分：主要内容和技术路线**
- 描述本文的核心内容、采用的技术方案和实现路线
- 篇幅约占摘要的三分之一

**第三部分：主要功能模块、成果和意义**
- 概括系统核心功能模块、最终成果及其应用价值与意义
- 篇幅约占摘要的三分之一

**字数建议**：中文摘要建议控制在页面约 2/3，英文摘要与中文内容对应

**禁止出现**：引用标记、图表编号、第一人称（我/我们）
**避免过度使用**：「本文」「本研究」开头句式

**摘要模板**：
```markdown
# 摘要

第一段：[研究背景]...[现实需求]...[本文目标]...（约150字）

第二段：[核心内容]...[技术方案]...[实现路线]...（约150字）

第三段：[功能模块]...[成果]...[应用价值与意义]...（约150字）

**关键词**：关键词1；关键词2；关键词3；关键词4；关键词5

---

# Abstract

Paragraph 1: Background, practical needs, and research objectives.

Paragraph 2: Core content, technical approach, and implementation roadmap.

Paragraph 3: Key functional modules, results, and significance.

**Keywords**: keyword1; keyword2; keyword3; keyword4; keyword5
```

### 4.2 引言

**内容要点**：
1. 研究背景（说明研究领域的现状）
2. 研究问题（指出存在的问题或空白）
3. 研究意义（说明研究的理论和实践价值）
4. 研究内容与方法（概述全文结构）

**字数占比**：占总字数的 10-15%

### 4.3 文献综述

**写作要点**：
1. 按主题或时间线组织文献
2. 不是简单罗列，要有分析评述
3. 指出现有研究的不足，引出本文贡献
4. 引用近 5 年文献占比 > 50%

**常见问题**：
- ❌ 「张三研究了A，李四研究了B，王五研究了C」（流水账式）
- ✅ 「关于XX问题，学界主要形成了两种观点。张三（2022）认为…，而李四（2023）则提出…。综合来看，现有研究多集中于…，而对…关注不足。」

### 4.4 正文章节

**论证逻辑**：
```
观点陈述 → 理论支撑/数据分析 → 案例说明 → 小结
```

**数据呈现**：
- 使用表格对比时，需有文字说明
- 图表需有标题和来源
- 数据需注明出处

**案例使用**：
- 案例应具有代表性
- 简明扼要，不要喧宾夺主
- 分析案例后要回扣论点

### 4.5 结论

**内容要点**：
1. 概括主要研究发现（呼应研究问题）
2. 理论贡献与实践启示
3. 研究局限性
4. 未来研究方向

**注意事项**：
- 不要提出新的论点
- 不要简单重复摘要
- 局限性分析要诚恳具体

## 五、语言润色要点

### 5.1 句长控制

**避免**：全文字数均匀（AI 特征）

**建议**：
- 简短句（10-15字）：强调、转折
- 中等句（15-30字）：陈述、说明
- 长句（30-50字）：阐述复杂观点

**目标分布**：句长标准差 > 10

### 5.2 标点使用

**增加多样性**：
- 适度使用冒号「：」引出说明
- 使用分号「；」并列列举
- 使用破折号「——」补充说明
- 使用问号「？」提出问题

### 5.3 过渡词使用

**避免 AI 高频过渡词**：
| 避免 | 可用但少量 | 推荐 |
|-----|----------|-----|
| 首先、其次、最后 | 第一、第二、第三 | 直接陈述 |
| 此外 | 另外 | 同时、与之相伴 |
| 综上所述 | 总体而言 | 总的来看 |
| 由此可见 | 这说明 | 这意味着 |

## 七、代码展示规范 ⭐

> 📌 **核心原则**：论文是"论述"而非"代码仓库"，代码展示必须精炼且有设计说明。

### 7.1 代码长度限制

| 规则 | 标准 | 说明 |
|------|------|------|
| 每段代码长度 | **≤ 20 行** | 超出用 `// ...省略...` 代替 |
| 禁止完整类展示 | ❌ | 不直接贴完整 Service、Controller 类 |
| 禁止 CRUD 展示 | ❌ | 通用的增删改查代码只描述不展示 |
| 关键代码优先 | ✅ | 只展示核心算法、关键逻辑、创新实现 |

### 7.2 代码展示三要素

每段代码必须包含：

```
1. 【设计说明】（代码前）：为什么这样做，解决什么问题
2. 【代码片段】（≤20行）：关键实现代码
3. 【效果分析】（代码后）：达到了什么效果，有什么优势
```

### 7.3 正确示例

```markdown
#### 5.2.1 用户认证模块实现

系统采用 JWT（JSON Web Token）实现无状态认证，避免服务端存储 Session 的开销。
用户登录成功后，服务端生成包含用户信息的 Token 返回给客户端，后续请求携带
Token 进行身份验证。

核心 Token 生成逻辑如下：

```java
public String generateToken(UserDetails userDetails) {
    Map<String, Object> claims = new HashMap<>();
    claims.put("username", userDetails.getUsername());
    claims.put("role", userDetails.getAuthorities());
    // ...省略其他声明设置...
    return Jwts.builder()
        .setClaims(claims)
        .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
        .signWith(SignatureAlgorithm.HS512, secretKey)
        .compact();
}
```

该方法通过 HS512 算法签名，设置了 24 小时的过期时间，既保证了安全性，
又避免了频繁登录的用户体验问题。经测试，Token 验证平均响应时间 < 10ms。
```

## 十八、常见问题速查

### 问题 1：代码堆砌

**表现**：直接贴完整 Service 类（50+ 行）

**解决**：
1. 精简为关键方法（≤20行）
2. 添加设计说明（为什么这样做）
3. 添加效果分析（达到了什么效果）

### 问题 2：图表缺失

**表现**：用 ASCII 字符画代替专业图表

**解决**：
1. 使用图表占位符标记位置
2. 输出 Mermaid 代码供转换
3. 使用 Draw.io/Visio 绘制

### 问题 3：缺少规定动作

**表现**：没有"国内外研究现状"、"可行性分析"

**解决**：
1. 按模板补充缺失章节
2. 确保三个维度都覆盖

### 问题 4：设计与实现混淆

**表现**：第4章直接贴代码，没有流程图

**解决**：
1. 第4章只讲设计（架构图、流程图、E-R图）
2. 第5章讲实现（代码、截图）
