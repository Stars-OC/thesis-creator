# 论文写作规范

论文撰写的具体规范，包括代码展示、图表占位符、学术写作风格。

---

## 代码展示规则

> **核心原则**：论文是"论述"而非"代码仓库"，代码展示必须精炼且有设计说明。

### 规则清单

| 规则 | 说明 |
|------|------|
| 代码长度限制 | 每段代码**不超过 20 行**，超出的用 `// ...省略...` 代替 |
| 设计说明(前) | 每段代码前必须有**设计动机说明**(为什么这样做) |
| 效果分析(后) | 每段代码后必须有**效果分析**(达到了什么效果) |
| 禁止完整类展示 | ❌ 禁止直接贴完整 Service 类、Controller 类 |
| 禁止 CRUD 展示 | ❌ 通用的 CRUD 代码不展示，只描述即可 |
| 关键代码优先 | ✅ 只展示核心算法、关键逻辑、创新实现 |

### 正确示例

```markdown
#### 5.2.1 用户认证模块实现

系统采用 JWT 实现无状态认证，避免服务端存储 Session 的开销。

核心 Token 生成逻辑如下：

```java
public String generateToken(UserDetails userDetails) {
    Map<String, Object> claims = new HashMap<>();
    claims.put("username", userDetails.getUsername());
    // ...省略其他声明设置...
    return Jwts.builder()
        .setClaims(claims)
        .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
        .signWith(SignatureAlgorithm.HS512, secretKey)
        .compact();
}
```

该方法通过 HS512 算法签名，设置了 24 小时的过期时间，既保证了安全性，
又避免了频繁登录的用户体验问题。
```

## 图表占位符规范

> **核心原则**：LLM 无法直接生成图片，必须使用**占位符机制**标记图表位置。

### 图表占位符格式

```markdown
<!-- 图表占位符：图4-1 系统架构图 -->
> 📊 **[图表占位符]**
> - **图表编号**：图4-1
> - **图表名称**：系统整体架构图
> - **图表类型**：架构图(建议使用 Draw.io / Visio 绘制)
> - **内容描述**：展示系统的分层架构...
> - **绘图要点**：
>   - 使用矩形表示各层
>   - 使用箭头表示调用关系
<!-- 图表占位符结束 -->
```

### 表格占位符格式

```markdown
<!-- 表格占位符：表4-1 用户表结构 -->
> 📋 **[表格占位符]**
> - **表格编号**：表4-1
> - **表格名称**：用户信息表(t_user)结构设计

| 字段名 | 类型 | 长度 | 允许空 | 主键 | 说明 |
|--------|------|------|--------|------|------|
| id | BIGINT | 20 | 否 | 是 | 用户ID，自增 |
| username | VARCHAR | 50 | 否 | | 用户名，唯一 |
<!-- 表格占位符结束 -->
```

## 学术写作风格

### AI 禁用词汇

- ❌ 「此外」「值得注意的是」「不可否认」「综上所述」
- ❌ 「首先...其次...最后...」(过度模板化)
- ❌ 「XXX是一种...，它具有...的特点」(教科书式)
- ✅ 「研究表明」「据统计分析」「笔者认为」「从实证角度来看」

### 必须包含的学术元素

| 元素 | 要求 | 示例 |
|------|------|------|
| 文献引用 | 关键论点必须有引用 | 「已有研究表明，RAG 技术能有效提升问答准确率[12]」 |
| 数据支撑 | 用数据论证观点 | 「系统响应时间从 500ms 优化至 80ms」 |
| 对比分析 | 技术选型需对比 | 「相较于 MySQL，PostgreSQL 在 JSON 查询方面具有优势」 |
| 方法论证 | 解释"为什么" | 「采用 Redis 缓存的主要原因是其高性能读写能力」 |
